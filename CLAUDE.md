## Commands

```bash
npm run dev              # start dev server (Turbopack)
npm run build             # production build (also runs the TypeScript check)
npm run lint               # ESLint (flat config, eslint.config.mjs)
npm run locations:import:kocaeli   # fetch+filter OSM/Overpass candidates for one province
npm run locations:verify           # polygon-verify, dedupe, score candidates -> data/locations/*.json
npm run locations:report           # print a verification summary from data/locations/*.json
npm run locations:test-filter      # unit-test the Overpass candidate filter logic
npm run test:photo-feature   # pure-logic tests for app/_lib/photo-validation.ts, no server needed
npm run test:photo-api       # E2E test of the /api/job-photos/process route against a running dev server
npm run test:photo-browser   # Playwright E2E for the photo-upload UI, against a running dev server
npm run test:regression      # Playwright smoke test that unrelated flows still work, against a running dev server
npm run test:location-system # Playwright E2E for the İl/İlçe/facility picker, against a running dev server
```

There is no Jest/Vitest — "tests" are plain Node scripts under `scripts/`, run directly (`node scripts/....mjs`), not through a runner. Two kinds:
- **Pure-logic** (`test-photo-feature.mjs`, `locations/test-filter.mjs`): import the `.ts` module directly using Node's built-in TypeScript support, no server needed.
- **Browser/E2E** (`test-photo-api.mjs`, `browser-test-*.mjs`): require `npm run dev` already running on `localhost:3000` first (`test-photo-api.mjs` hits the route with `fetch`; the `browser-test-*.mjs` ones drive real Chromium via Playwright). `test:photo-browser` needs fixture images from `node scripts/generate-photo-test-fixtures.mjs` first.

Treat `npm run lint` + `npm run build` as the baseline correctness gate for app code; use the scripts above (or a throwaway `node -e "..."`) to independently cross-check pure logic (money parsing, phone normalization, geo functions, photo validation) before trusting it.

`scripts/tmp-*.mjs` (e.g. `tmp-completion-flow-test.mjs`, `tmp-withdraw-offer-test.mjs`) are one-off Playwright verification scripts written alongside a specific feature commit and left committed afterward as a record — they are not npm scripts and aren't meant to be maintained as a regression suite. Follow the same pattern for a new feature (`tmp-<feature>-test.mjs`, requires `npm run dev` running) rather than editing an old one to test something unrelated.

## Architecture

**Stack:** Next.js 16 (App Router, Turbopack), TypeScript, React 19, Tailwind CSS v4. No auth provider, no persistent server-side state for the actual app — nearly everything is client-side `localStorage`, written as typed data-layer modules under `app/_lib/`. There are two Route Handlers: `app/api/job-photos/process/route.ts` (stateless image processing that can't run in the browser; see "Photo upload pipeline" below) and `app/api/health/route.ts` (a DB connectivity check — see "Database scaffold" below). Both are the only server-side code in the app; every feature (`users`, `job-store`, `offers`, `session`, `notifications`, ...) still reads/writes `localStorage` exclusively.

### Directory roles

- `app/_components/` — UI components (route-excluded via the `_` prefix). Presentational components have no hooks; anything reading session/localStorage is a `"use client"` component.
- `app/_lib/` — the entire data/business layer: localStorage-backed "tables" (`session.ts`, `users.ts`, `job-store.ts`, `offers.ts`), an IndexedDB blob store (`photo-blob-store.ts`), pure validation modules (`*-form-validation.ts`, `money.ts`, `phone.ts`, `password-rules.ts`, `photo-validation.ts`), a shared cross-source lookup (`jobs-lookup.ts`), and reactive hooks (`use-*.ts`). Four of the `use-*.ts` hooks are reusable UI-interaction primitives rather than data hooks — reuse them instead of reimplementing the same behavior locally: `use-dismiss-animation.ts` (measure element height → animate slide+fade+collapse → invoke a callback only after the animation completes, skipped entirely under `prefers-reduced-motion`; used by the notification list), `use-scroll-lock.ts` (iOS/Android-safe body scroll lock via `position: fixed` + scroll-offset restore, not plain `overflow: hidden`; used by the mobile nav menu), `use-dropdown.ts` (open/close state + outside-click/Escape-to-close, with an `extraRefs` escape hatch for portal-rendered content that `Node.contains()` would otherwise see as "outside"; used by the mobile menu and profile/notification dropdowns), and `use-auto-dismiss-banner.ts` (time-triggered, not user-triggered: shows a standalone banner for a fixed duration then fades it out — deliberately separate from `use-dismiss-animation.ts`'s click-triggered slide+collapse; used by the job-requests panel's delete confirmation banner).
- `app/api/job-photos/process/` — `runtime = "nodejs"` (needs `sharp`'s native bindings, won't run on Edge). Stateless request-in/image-out; writes nothing server-side.
- `app/_data/turkey/` — static reference JSON for provinces and districts only (`provinces.json`, `districts.json`, complete for all 81 provinces), bundled into the app at build time. Facility data (OSB/port/free-zone/etc.) does **not** live here — see "Turkey location data" below.
- `data/` (repo root, **not** under `app/`) — output of the location-verification pipeline: `data/locations/locations.json` (the single facility dataset), `data/locations/location_suggestions.json`, `data/locations/location_manual_overrides.json`, `data/locations/raw/*.json` (per-province Overpass fetch cache), and `data/boundaries/*.geojson` (district polygons, currently only Kocaeli). Read by `app/_lib/turkey-locations.ts` via a relative import; regenerated by the scripts below, not hand-edited.
- `scripts/locations/` — standalone Node ESM (`.mjs`, no TypeScript, no ts-node) CLI pipeline: Overpass API fetch → district-polygon verification → dedup → confidence scoring → writes to `data/locations/`. Not part of the Next.js build; run manually via the `locations:*` npm scripts. `migrate-legacy.mjs` is a one-off (already-run) script, not an npm script — see "Turkey location data" below.

### No real backend: what that means in practice

- "Tables" are localStorage-backed modules, not a database. Reads/writes go through a cache-plus-`useSyncExternalStore` pattern (see `session.ts`, `job-store.ts`, `offers.ts`): a module-level cache compares the raw localStorage string to avoid returning new object references on every render, `getServerSnapshot` always returns `null`/`[]` for SSR safety, and writes call an internal `notify()` so every subscribed component re-renders.
- **Do not use `useEffect` + `setState` to read localStorage on mount** — the project's ESLint config enforces `react-hooks/set-state-in-effect` and will fail lint on that pattern. Use the existing `useSyncExternalStore`-based hooks (`use-session.ts`, `use-jobs.ts`, `use-offers.ts`) or extend that pattern for new localStorage-derived state.
- Passwords (`app/_lib/users.ts`) are hashed with unsalted SHA-256 via the browser's `crypto.subtle` — explicitly documented in that file as a dev-only stand-in, not production-grade auth.
- Three dev-only seed accounts (Zeynep/hizmet-alan, Mert/hizmet-veren, Mehmet Demir/hizmet-veren — see `DEV_ACCOUNTS` in `users.ts`) are created by `seedDevAccountsIfNeeded()`, called from `login-form.tsx`. The gate is an allow-list, `NODE_ENV === "development"` (not a `!== "production"` block-list) specifically so it stays off under `next build && next start`, including Vercel preview/production.
- Authorization checks (role checks, ownership checks, offer-status transitions) are implemented once in the `_lib` data-layer functions (e.g. `offers.ts#createOffer`/`updateOfferStatus`) and called from UI — never re-implemented ad hoc in a component. `contact-access.ts#getRevealedContactForOffer` is the single gate for revealing a counterparty's phone/email (only when an offer is `accepted` and the caller is a party to it); nothing else should read raw phone/email off a `StoredUser`.
- The `/api/job-photos/process` route reads a client-supplied `role` field to gate uploads to `hizmet-alan` — this is defense in depth, not the authorization boundary. Since there's no real session on the server, a forged request could send any role; the actual authority check happens again server-side-equivalent in `job-store.ts#createJob`/`deleteJobPhoto` (the localStorage write path), which is what a spoofed request can't bypass.
- `reset-demo-data.ts` + `app/gelistirme/demo-veri-sifirla` (`notFound()` unless `NODE_ENV === "development"`) is a dev-only tool to wipe transactional data (jobs/offers/ratings/notification read+dismiss state) belonging to the seed accounts (`DEV_ACCOUNT_EMAILS`) without touching the seed accounts themselves — a job counts as the seed account's if `requesterId` matches, an offer/rating if either it belongs to such a job *or* its `providerId`/`(rater/provider)Id` is a seed account (so a seed provider's offer on a *real* user's job is removed without touching that job or its other offers). It has a manual dry-run-then-apply mode (`planDemoDataReset()` / `executeDemoDataReset()`, used by that page) and a one-time auto-migration (`runDemoDataResetMigrationIfNeeded()`) gated by a `localStorage` version flag — bump `DEMO_DATA_RESET_VERSION` to make it re-run once in browsers that already completed an earlier version.

### Photo upload pipeline

Job photos flow through three layers that only make sense together:

1. **Client-side pre-check** (`photo-validation.ts`): sniffs the real file format from magic bytes (never trusts the extension/declared MIME), enforces size/count limits, and SHA-256-hashes content to reject exact duplicates.
2. **Server-side processing** (`app/api/job-photos/process/route.ts`, Node runtime): re-validates by actually decoding the image with `sharp`. HEIC/HEIF (common from iPhones) can't be decoded by the bundled libvips due to codec licensing, so those go through `heic-convert` (WASM, unrestricted) first. The pipeline then auto-rotates from EXIF orientation and **strips all EXIF/GPS metadata** (no `.withMetadata()` call) before returning JPEG/PNG/WebP bytes.
3. **Client-side storage** (`photo-blob-store.ts`): the processed blob is written to IndexedDB (localStorage's ~5-10MB quota can't hold up to 10 photos × 10MB each). `job-store.ts` only stores a `storageKey` pointing into that blob store on the `Job.photos` record; `use-job-photo-url.ts` resolves a `storageKey` back to a revocable `URL.createObjectURL` for `<img>` rendering.

### Domain model

- Two roles: `hizmet-alan` (service requester, posts `Job`s) and `hizmet-veren` (service provider, posts `Offer`s on jobs). Role gating is checked both in UI (which fields/buttons render) and in the data layer (which operations are allowed). Registration (`login-form.tsx` kayıt mode, validated by `register-form-validation.ts`) collects company info for both roles — `companyName`, `companyType` (`company-type.ts`), `province`/`district`, plus mandatory KVKK/terms checkboxes — stored directly on `StoredUser` (`users.ts#registerUser`); this is separate from and unrelated to the richer, optional `ProviderProfile` below.
- Jobs come from two separate sources merged only through `jobs-lookup.ts#findJobById`: `jobs.ts` is a static in-memory array of seed example listings (`requesterId: null`, not localStorage-backed), while `job-store.ts` is the real localStorage "table" for user-created jobs. Never query `job-store.ts` alone when you need "any job by id" — other modules (`offers.ts`, `contact-access.ts`) go through `jobs-lookup.ts` so seed listings resolve too.
- `Job.requesterId` links a job to the user who created it (`null` for the seeded example listings) — this is what `contact-access.ts` and the notifications system use to determine "is this my job."
- `Job.status` (`"yayinda" | "tamamlandi" | "iptal"`) never changes after creation for a real user's job — `createJob` always writes `"yayinda"`, and no other function (including `updateJob`) touches it. Time-varying state (open/closed to new offers, in-progress, etc.) is derived at read-time from the job's `Offer`s via helpers in `job-requests.ts` (`jobHasAcceptedOffer`, `getJobRequestFilter`, `getJobOfferAvailability`, `getJobAvailabilityForProvider`) — check that file, not `Job.status`, when you need a job's effective state.
- **Job editing** is gated by `job-requests.ts#isJobEditable`/`JOB_NOT_EDITABLE_MESSAGE` (the offer process having started — any offer in `ENGAGED_OFFER_STATUSES` ∪ `completed` ∪ `cancelled` — permanently locks a job from further edits) — the single source of truth called from three independent layers that must stay in sync: the "İlanı Düzenle" link's visibility (`job-requests-panel.tsx`), the edit route's own guard (`job-edit-form.tsx`), and the data-layer write (`job-store.ts#updateJob`, which takes the current `Offer[]` as a parameter precisely so it can re-check this without importing `offers.ts` — that would be circular, since `offers.ts` already imports `job-store.ts`).
- `Offer.status`: `pending -> accepted | rejected` via `offers.ts#updateOfferStatus` (job-owner/requester only). An `accepted` offer can further move to `in_progress` via `offers.ts#startWorkForOffer`, or to `agreement_failed` via `offers.ts#recordAgreementFailure` (both requester-only, both leave `Job.status` untouched). `agreement_failed` is deliberately excluded from `jobHasAcceptedOffer`, which is what makes the job re-offerable again. A `pending` offer can also move sideways to `withdrawn` via `offers.ts#withdrawOffer` (provider-only) — deliberately excluded from `ENGAGED_OFFER_STATUSES` too, since a never-accepted offer never held capacity or contact access.
- **Single active acceptance per job**: a job can receive `pending` offers from multiple providers concurrently, but at most one may ever be settled at a time — enforced by `job-requests.ts#isOfferPendingActionBlocked` (built on `getSettledOfferForJob`, which unions `getEngagedOfferForJob` and `getCompletedOfferForJob` — the single shared source of truth), called from *both* `offers.ts#updateOfferStatus` (rejects a second accept/reject attempt at the data layer, immune to a stale tab or a bypassed button) and `incoming-offer-card.tsx` (swaps the Kabul Et/Reddet buttons for a fixed notice, `job-requests.ts#OFFER_PENDING_BLOCKED_MESSAGE`). Deliberately does **not** mutate sibling offers to `rejected` — they stay `pending` untouched, so if the settled offer later fails out to `agreement_failed` they become actionable again automatically (purely derived, no extra write); once the settled offer instead progresses *past* `accepted` (work actually starts), those siblings surface as closed in the provider's own tabs instead — see "Provider offer tabs" below. Separately-scoped gap: `jobHasAcceptedOffer` (unlike `getSettledOfferForJob`) still excludes `completed`, so `getJobOfferAvailability`/`createOffer` continue to treat a job as open once its winning offer is fully `completed` — a brand-new provider who never bid can still see "Teklife Açık" and submit a doomed offer there.
- **Provider offer tabs ("Verdiğim Teklifler")**: `job-requests.ts#getProviderOfferFilter(offer, offers)` — note it takes the *full* offers array, not just the caller's own, because it needs sibling context — is the single source of truth for which of the four `my-offers-panel.tsx` tabs (Aktif/Devam Eden/Tamamlanan/Kapanan Teklifler) an offer appears in, and `getProviderOfferNotificationHref`/`getProviderOffersTabHref` build every Hizmet Veren notification's `href` from that same function, so a notification always lands on the tab the offer actually renders in (never a hand-rolled route string). "Kapanan Teklifler" bundles three unrelated reasons an offer is permanently non-actionable: (1) a still-`pending` sibling offer on a job where a *different* offer has progressed past `accepted` (`isOfferClosedByJobProgress`, built on `getSettledOfferForJob` rather than a new status) — shown with an overridden "Başka Bir Hizmet Verenle Anlaşıldı" badge and a dedicated notification (see below); (2) `agreement_failed`; (3) `cancelled` (a disputed job resolved as cancelled, shown with an extra "İtiraz sonrası iş iptal edildi." line). (2) and (3) keep their real `getOfferStatusLabel` badge instead of the override. None of the three render any action button in that tab.
- **Completion is a two-party handshake, not a single write.** From `in_progress`, the provider calls `offers.ts#requestCompletion` (-> `completion_requested`, records `completionRequestedByUserId`/`completionRequestedAt`); the requester then either `confirmCompletion` (-> `completed`) or `disputeCompletion` with a 10–1000 char note (-> `completion_disputed`), and a disputed offer is resolved by the requester via `resolveCompletionDispute` to either `completed` or `cancelled`. The party that requested completion is blocked from confirming/disputing their own request. Because there's no server-side scheduler, a `completion_requested` offer that the requester never acts on is auto-approved after `COMPLETION_AUTO_APPROVE_DAYS` (7 days) by `applyExpiredCompletionAutoApprovals()` — this runs lazily (checked from `use-offers.ts` on read, not on a timer) and sets `autoCompleted: true`, which caps the `ratings.ts` review window to `AUTO_COMPLETED_RATING_WINDOW_DAYS` (30 days; manually-confirmed completions have no review deadline).
- **Provider capacity**: `provider-capacity.ts#MAX_ACTIVE_JOBS` (5) caps how many jobs a `hizmet-veren` can hold concurrently, counted via `getActiveJobCount`/`hasReachedActiveJobLimit` over `job-requests.ts#ENGAGED_OFFER_STATUSES` (`accepted`, `in_progress`, `completion_requested`, `completion_disputed`) — a completion request or an unresolved dispute still counts against capacity; only a terminal status (`completed`, `cancelled`, `agreement_failed`, `rejected`, `withdrawn`) frees a slot.
- **Ratings** (`ratings.ts`): a requester submits one 1–5 star `Rating` per completed `Offer` for the provider (`submitRating`), aggregated per-provider via `getProviderRatingSummary`. Gated by the auto-completion review window above, not by role/ownership checks alone.
- **Provider profiles** (`provider-profile.ts`): an optional `ProviderProfile` (company name, logo, 50–500 char bio, founded year, service regions, expertise, `serviceCategories`, `serviceFeatures`, `experienceRange`) on `StoredUser`, meaningful only for `hizmet-veren` — fully optional and normalized on read like `Job.photos`, so pre-migration users just render blank/`"-"` fields rather than fabricated data. **Two separate, deliberately-independent edit surfaces write to the same `StoredUser.providerProfile` object** — each preserves the other's fields on save rather than clobbering them: Hesap Ayarları > Firma Profili (`provider-profile-editor.tsx`, requires companyName+bio) through `users.ts#updateProviderProfile` (carries `serviceCategories`/`serviceFeatures`/`experienceRange` through unchanged), and Panel > Profilim > Hizmet Bilgilerim (`service-info-editor.tsx`, a completion flow that never requires companyName/bio) through `users.ts#updateProviderServiceInfo` (carries companyName/bio/foundedYear/logoStorageKey/expertise through unchanged). `regions` is the one field both screens genuinely share and either can overwrite. The logo blob lives in the same IndexedDB store as job photos (`photo-blob-store.ts`) under a separate key namespace, uploaded client-side only (not via `/api/job-photos/process`, which stays `hizmet-alan`-gated). Viewed read-only by the counterparty inline within `IncomingOfferCard` (no separate drawer/modal component) — it never reads phone/email/address/tax/TC fields; contact reveal remains solely `contact-access.ts#getRevealedContactForOffer`'s job. `getProviderProfileSummary` computes average rating, in-progress count, and acceptance rate purely from existing `ratings.ts`/`job-requests.ts` data — no separate event log, and no "average response time" metric (dropped: providers initiate offers here, so there's no "provider responded" event to time).
- **Service catalog** (`service-catalog.ts`): the platform's single centralized, hierarchical service taxonomy (`SERVICE_CATEGORY_GROUPS`) — feeds *both* a provider's multi-select "Hizmet Bilgilerim" service selection *and* the job-posting/editing "Hizmet Kategorisi" single select (`job-request-form.tsx`, `job-edit-form.tsx`), sharing the same `id` space so a provider's selected service and a job's category can in principle line up (no automatic matching yet — just a shared data foundation). Supersedes but does not remove `jobs.ts#SERVICE_CATEGORIES` (an old flat 8-item list still written by some legacy records); the two are bridged by a one-way, best-effort migration layer — `resolveLegacyJobCategoryToId` (single value, for `Job.category`), `migrateLegacyExpertiseToServiceCategoryIds` (multi-value, for `ProviderProfile.expertise`), and `getCategoryDisplayLabel` (safe display for either old or new values). `"Depolama"` is deliberately left unmapped (no 1:1 equivalent in the new catalog) — migration functions silently skip values they can't map rather than inventing one; original legacy data is never mutated or lost. Also houses `SERVICE_FEATURE_OPTIONS`/`EXPERIENCE_RANGE_OPTIONS` (Aşama 2 additions to `ProviderProfile`) and `getProviderServiceInfoCompletion` (the "Profil Tamamlanma %" checklist shown in `service-info-editor.tsx`).

### Notifications are derived, not stored

`notifications.ts` computes each user's `AppNotification[]` live from the current `Offer.status` (plus `Offer.createdAt`) every render — there is no separate notifications/events table. This is why notification ids stay stable (derived from `offer.id` + event name) and why read/dismissed state (tracked separately, see below) keeps matching up correctly across sessions. One consequence to keep in mind when adding a new offer-status transition: notifications tied to a **transient** status (e.g. `accepted`, `completion_requested`) disappear on their own the moment the offer moves past that status, because the derivation only ever looks at the offer's *current* status; only notifications tied to a **terminal** status (`completed`, `rejected`, `cancelled`, `agreement_failed`, `withdrawn`) persist. Requesters and providers each get their own distinct set of `NotificationType`s derived from the same offers — never the other role's. Almost every notification filters `myOffers` by the viewer's *own* `offer.status`; the one exception is `baska_hizmet_verenle_anlasildi` ("Başka Bir Hizmet Verenle Anlaşıldı"), which instead filters the viewer's still-`pending` offers by whether a *sibling* offer on the same job has closed them (`getProviderOfferFilter(offer, offers) === "kapanan-teklifler"`) and reads that sibling's `updatedAt` for `createdAt` — see "Provider offer tabs" above. Read state (`notification-reads.ts`) and dismissed state (`notification-dismissals.ts`) are two separate per-user localStorage keys (`malsevk_read_notifications_<userId>` / equivalent for dismissals) — intentionally not cleared by `clearSession()`, so logging out and back in doesn't reset what a user has already seen or dismissed.

### Turkey location data

`app/_lib/turkey-locations.ts` reads facility data (OSB/port/free-zone/depot/etc.) from a **single source**, `data/locations/locations.json` — there is no separate hand-curated facility list anywhere else in the repo. That one file mixes two kinds of records under a shared schema (`id, name, type, provinceId, districtId, aliases, active`):

1. **Pipeline-verified entries** — provinces run through `locations:import:<province>` + `locations:verify` (currently only Kocaeli, `provinceCode: "41"`), scored via real Overpass/OSM polygon verification.
2. **Migrated legacy entries** — every other province, carried over verbatim by the one-off `scripts/locations/migrate-legacy.mjs` from the old hand-curated `app/_data/turkey/{osb,ports,free-zones,logistics-centers}.json` files (which no longer exist in the repo; that script deleted them after migrating). These are `active: true` for display purposes but have **not** been through real polygon verification — don't assume "present in `locations.json`" implies "geo-verified" for any province other than Kocaeli.

Only `active === true` records are ever shown; a `districtId: null` record (district unresolved) deliberately never matches any district filter rather than risk being shown under the wrong one. Lookups are always **province + district + facility type together** via `getFacilitiesByProvinceDistrictAndType(provinceId, districtId, type)` — there's no province-only or name-only lookup. `app/_data/turkey/{provinces,districts}.json` (81/81 provinces) are the unrelated, complete, static province/district reference and are not part of this pipeline.

### Database scaffold

`app/_lib/db.ts` (lazy Neon client via `DATABASE_URL`) and `drizzle.config.ts` exist but are **not wired into any business logic** — their only consumer is `app/api/health/route.ts`, which runs `SELECT 1` and returns 503 if it fails or `DATABASE_URL` is unset. No `db/schema.ts` exists yet and no migrations have been generated. Every real feature still reads/writes `localStorage` exclusively (see "No real backend" above) — don't assume this scaffold means any given feature is backed by a real database.

### Design tokens

Brand colors/spacing live as CSS custom properties in `app/globals.css` (`:root` raw values → `@theme inline` Tailwind v4 token mapping, e.g. `--color-primary`, `--color-accent`, `--color-success`, `--radius-card`). Add new colors there rather than hardcoding hex in components.
