## Commands

```bash
npm run dev              # start dev server (Turbopack)
npm run build             # production build (also runs the TypeScript check)
npm run lint               # ESLint (flat config, eslint.config.mjs)
npm run locations:import:kocaeli   # fetch+filter OSM/Overpass candidates for one province
npm run locations:verify           # polygon-verify, dedupe, score candidates -> data/locations/*.json
npm run locations:report           # print a verification summary from data/locations/*.json
npm run locations:test-filter      # unit-test the Overpass candidate filter logic
npm run test:photo-feature   # pure-logic tests for app/_lib/photo-validation.ts, no server needed
npm run test:photo-api       # E2E test of the /api/job-photos/process route against a running dev server
npm run test:photo-browser   # Playwright E2E for the photo-upload UI, against a running dev server
npm run test:regression      # Playwright smoke test that unrelated flows still work, against a running dev server
npm run test:location-system # Playwright E2E for the İl/İlçe/facility picker, against a running dev server
```

There is no Jest/Vitest — "tests" are plain Node scripts under `scripts/`, run directly (`node scripts/....mjs`), not through a runner. Two kinds:
- **Pure-logic** (`test-photo-feature.mjs`, `locations/test-filter.mjs`): import the `.ts` module directly using Node's built-in TypeScript support, no server needed.
- **Browser/E2E** (`test-photo-api.mjs`, `browser-test-*.mjs`): require `npm run dev` already running on `localhost:3000` first (`test-photo-api.mjs` hits the route with `fetch`; the `browser-test-*.mjs` ones drive real Chromium via Playwright). `test:photo-browser` needs fixture images from `node scripts/generate-photo-test-fixtures.mjs` first.

Treat `npm run lint` + `npm run build` as the baseline correctness gate for app code; use the scripts above (or a throwaway `node -e "..."`) to independently cross-check pure logic (money parsing, phone normalization, geo functions, photo validation) before trusting it.

## Architecture

**Stack:** Next.js 16 (App Router, Turbopack), TypeScript, React 19, Tailwind CSS v4. No database, no auth provider, no persistent server-side state — nearly everything is client-side `localStorage`, written as typed data-layer modules under `app/_lib/`. The one exception is a single **stateless** Route Handler (`app/api/job-photos/process/route.ts`) used for image processing that can't run in the browser; see "Photo upload pipeline" below.

### Directory roles

- `app/_components/` — UI components (route-excluded via the `_` prefix). Presentational components have no hooks; anything reading session/localStorage is a `"use client"` component.
- `app/_lib/` — the entire data/business layer: localStorage-backed "tables" (`session.ts`, `users.ts`, `job-store.ts`, `offers.ts`), an IndexedDB blob store (`photo-blob-store.ts`), pure validation modules (`*-form-validation.ts`, `money.ts`, `phone.ts`, `password-rules.ts`, `photo-validation.ts`), a shared cross-source lookup (`jobs-lookup.ts`), and reactive hooks (`use-*.ts`).
- `app/api/job-photos/process/` — the only Route Handler in the app. `runtime = "nodejs"` (needs `sharp`'s native bindings, won't run on Edge). Stateless request-in/image-out; writes nothing server-side.
- `app/_data/turkey/` — static reference JSON (provinces, districts, OSB/port/free-zone/logistics-center facility lists) bundled into the app at build time.
- `data/` (repo root, **not** under `app/`) — output of the location-verification pipeline: `data/locations/{locations,location_aliases,location_sources,location_suggestions}.json` and `data/boundaries/*.geojson`. Read by `app/_lib/turkey-locations.ts` via a relative import; regenerated by the scripts below, not hand-edited.
- `scripts/locations/` — standalone Node ESM (`.mjs`, no TypeScript, no ts-node) CLI pipeline: Overpass API fetch → district-polygon verification → dedup → confidence scoring → writes to `data/locations/`. Not part of the Next.js build; run manually via the `locations:*` npm scripts.

### No real backend: what that means in practice

- "Tables" are localStorage-backed modules, not a database. Reads/writes go through a cache-plus-`useSyncExternalStore` pattern (see `session.ts`, `job-store.ts`, `offers.ts`): a module-level cache compares the raw localStorage string to avoid returning new object references on every render, `getServerSnapshot` always returns `null`/`[]` for SSR safety, and writes call an internal `notify()` so every subscribed component re-renders.
- **Do not use `useEffect` + `setState` to read localStorage on mount** — the project's ESLint config enforces `react-hooks/set-state-in-effect` and will fail lint on that pattern. Use the existing `useSyncExternalStore`-based hooks (`use-session.ts`, `use-jobs.ts`, `use-offers.ts`) or extend that pattern for new localStorage-derived state.
- Passwords (`app/_lib/users.ts`) are hashed with unsalted SHA-256 via the browser's `crypto.subtle` — explicitly documented in that file as a dev-only stand-in, not production-grade auth.
- Two dev-only seed accounts (Zeynep/hizmet-alan, Mert/hizmet-veren) are created by `seedDevAccountsIfNeeded()`, gated on `process.env.NODE_ENV !== "production"`.
- Authorization checks (role checks, ownership checks, offer-status transitions) are implemented once in the `_lib` data-layer functions (e.g. `offers.ts#createOffer`/`updateOfferStatus`) and called from UI — never re-implemented ad hoc in a component. `contact-access.ts#getRevealedContactForOffer` is the single gate for revealing a counterparty's phone/email (only when an offer is `accepted` and the caller is a party to it); nothing else should read raw phone/email off a `StoredUser`.
- The `/api/job-photos/process` route reads a client-supplied `role` field to gate uploads to `hizmet-alan` — this is defense in depth, not the authorization boundary. Since there's no real session on the server, a forged request could send any role; the actual authority check happens again server-side-equivalent in `job-store.ts#createJob`/`deleteJobPhoto` (the localStorage write path), which is what a spoofed request can't bypass.

### Photo upload pipeline

Job photos flow through three layers that only make sense together:

1. **Client-side pre-check** (`photo-validation.ts`): sniffs the real file format from magic bytes (never trusts the extension/declared MIME), enforces size/count limits, and SHA-256-hashes content to reject exact duplicates.
2. **Server-side processing** (`app/api/job-photos/process/route.ts`, Node runtime): re-validates by actually decoding the image with `sharp`. HEIC/HEIF (common from iPhones) can't be decoded by the bundled libvips due to codec licensing, so those go through `heic-convert` (WASM, unrestricted) first. The pipeline then auto-rotates from EXIF orientation and **strips all EXIF/GPS metadata** (no `.withMetadata()` call) before returning JPEG/PNG/WebP bytes.
3. **Client-side storage** (`photo-blob-store.ts`): the processed blob is written to IndexedDB (localStorage's ~5-10MB quota can't hold up to 10 photos × 10MB each). `job-store.ts` only stores a `storageKey` pointing into that blob store on the `Job.photos` record; `use-job-photo-url.ts` resolves a `storageKey` back to a revocable `URL.createObjectURL` for `<img>` rendering.

### Domain model

- Two roles: `hizmet-alan` (service requester, posts `Job`s) and `hizmet-veren` (service provider, posts `Offer`s on jobs). Role gating is checked both in UI (which fields/buttons render) and in the data layer (which operations are allowed).
- `Job.requesterId` links a job to the user who created it (`null` for the seeded example listings) — this is what `contact-access.ts` and the notifications system use to determine "is this my job."
- `Offer.status`: `pending -> accepted | rejected`, transitioned only via `offers.ts#updateOfferStatus`, only by the requester who owns the underlying job.

### Turkey location data

`app/_lib/turkey-locations.ts` merges two sources of facility data (OSB/port/free-zone/logistics-center/etc.) for the "İşin Yapılacağı Yer" cascading picker:

1. **Legacy hand-curated entries** in `app/_data/turkey/{osb,ports,free-zones,logistics-centers}.json` — still the only source for provinces that haven't been through the verification pipeline.
2. **Pipeline-verified entries** from `data/locations/locations.json`, filtered to `active === true && verification_status === "VERIFIED"` only. Records go through `REVIEW`/`REJECTED`/`OUTDATED` states and a 0–100 `confidence_score`; nothing below `VERIFIED` is ever shown to end users.

When a province has been run through `locations:import:<province>` + `locations:verify`, its legacy entries in `app/_data/turkey/*.json` are removed (superseded) rather than left duplicated — see how Kocaeli (`provinceCode: "41"`) was stripped out of those four files. Province/district data itself (`app/_data/turkey/{provinces,districts}.json`) is complete for all 81 provinces and is not part of this pipeline.

Facility lookups are always **province + district**, never province alone — `getFacilitiesByProvinceAndDistrict()` in `turkey-locations.ts` normalizes Turkish casing (`toLocaleLowerCase("tr-TR")`) before comparing, and a `district: null` record deliberately never matches any district filter rather than risk being shown under the wrong one.

### Design tokens

Brand colors/spacing live as CSS custom properties in `app/globals.css` (`:root` raw values → `@theme inline` Tailwind v4 token mapping, e.g. `--color-primary`, `--color-accent`, `--color-success`, `--radius-card`). Add new colors there rather than hardcoding hex in components.
